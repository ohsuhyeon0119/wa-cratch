# sw-brainstorming

`superpowers:brainstorming` 스킬을 기반으로 동작한다.
아래는 프로젝트에 맞게 추가된 규칙이다.

## 입력 형식

이슈의 구조를 인식한다:
- **개요**: 기능 아이디어 (큰 틀)
- **유저 시나리오**: Given / When / Then
- **기술 힌트**: 선택적 힌트

## 기존 디자인 확인 (최우선)

브레인스토밍 시작 전, `design/` 디렉토리에 관련 HTML 파일이 있는지 먼저 확인한다.

- 해당 페이지의 디자인 파일이 존재하면 **UI 구조·레이아웃·컴포넌트 명칭은 그대로 따른다**
- 디자인 파일이 다루지 않는 부분(인터랙션, 상태 관리, API 연동 방식 등)은 브레인스토밍에서 자유롭게 논의한다
- 디자인에 명시된 내용과 다르게 구현하려면 반드시 개발자 승인을 먼저 받는다

## 브레인스토밍 진행 방식

1. 이슈의 개요와 유저 시나리오를 읽고 불명확한 부분 파악
2. 개발자에게 질문 (한 번에 하나씩)
3. 답변 기반으로 설계 구체화
4. 설계 내용을 섹션별로 제시 → 개발자 확인
5. 반복 (개발자가 승인할 때까지)

## GitHub Wiki 업데이트

브레인스토밍 결과가 기존 기획 문서와 충돌하거나 새로운 결정이 생기면:
- 변경이 필요한 Wiki 페이지와 내용을 개발자에게 먼저 제시
- 개발자 승인 후 `mcp__github__update_wiki_page`로 업데이트
- 업데이트 완료 후 다음 단계 진행

## 완료 조건

개발자가 설계에 최종 승인하면 `superpowers:sw-tdd` 스킬로 넘어간다.
승인 없이 구현 단계로 넘어가지 않는다.
